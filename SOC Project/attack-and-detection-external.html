Attack & Detection

Overview

In this phase, multiple attack techniques were executed against the lab environment to evaluate detection and response capabilities.

  • External attacks (Kali Linux)
  • Validation of SIEM, IDS, and endpoint detection

Methodology:
Each attack follows the same structure: description → execution → evidence → detection → result → analysis

Lab Environment Note:
Due to hardware resource limitations, not all virtual machines in the lab environment could be executed simultaneously during this phase. As a result, only the systems relevant to each specific attack scenario were powered on.

This approach ensured optimal performance and stability during attack simulations, while still maintaining realistic testing conditions for detection and analysis.

Nmap Scan (Network Reconnaissance)

Description

An Nmap SYN scan was performed from the Kali Linux attacker machine to enumerate hosts, open ports, and services across the internal network.

Execution

  • Command used: nmap -sS -sV 192.168.59.0/24
  • Source machine: Kali Linux
  • Target: Internal network (192.168.59.0/24)

Command Breakdown

  • -sS (SYN Scan): Performs a half-open TCP scan by sending SYN packets without completing the full handshake. This allows fast and relatively stealthy detection of open ports.
  • -sV (Service Version Detection): Identifies running services and their versions by analyzing responses from open ports. This helps determine potential vulnerabilities.
  • 192.168.59.0/24: Scans the entire subnet to discover all active hosts and exposed services within the network.

This scan reveals the attack surface by identifying reachable hosts, open ports, and running services, forming the foundation for further exploitation attempts.

Execution Evidence

Nmap scan results - host discovery part 2

Execution Findings

  • Domain Controller (DC01) identified with Kerberos, LDAP, SMB services
  • Windows host exposing RPC and SMB services
  • SIEM01 and server identified with SSH (port 22) open
  • WEB01 identified with http service (port 80)

Logs Generated

  • Network IDS alerts (Suricata)
  • Wazuh SIEM alerts

Detection Evidence

Wazuh alert - Nmap detection

Detection

  • Detected by: Wazuh SIEM
  • Rule: Custom Nmap SYN Scan detection
  • MITRE ATT&CK: T1595 – Reconnaissance
  • Severity: 3
  • Action: Allowed (no blocking)

Result

  • Logs generated: ✔
  • Detected: ✔
  • Blocked: ❌

Analysis:
The Nmap scan successfully identified multiple critical systems within the network, including the domain controller, web server, file server and SIEM server. The activity was detected by a custom Wazuh rule, confirming visibility into reconnaissance techniques.

However, the scan was not blocked, highlighting a gap between detection and prevention. This indicates that while monitoring is effective, additional controls such as firewall rules or intrusion prevention systems (IPS) are required to actively mitigate reconnaissance activity.

The scan results were later used to identify attack surfaces, including SSH access on the SIEM server, which was targeted in a subsequent brute-force attack.

SSH Brute Force Attack (SIEM01)

Description

A brute-force attack was conducted against the SSH service on the SIEM01 server to simulate unauthorized access attempts using a password wordlist.

Execution

  • Tool used: Hydra
  • Command: hydra -l siemadmin -P passwords.txt ssh://192.168.59.40
  • Source machine: Kali Linux
  • Target: SIEM01 (SSH)

Command Breakdown

  • hydra: A password brute-force tool used to automate authentication attempts against various network services.
  • -l siemadmin: Specifies the target username for the attack.
  • -P passwords.txt: Provides a wordlist of passwords used to attempt authentication.
  • ssh://192.168.59.40: Targets the SSH service on the SIEM server.

Phase 1 — Failed Attempt (Incorrect Username)

  • Username used: root
  • Password list: passwords.txt
  • Result: Authentication failed

Execution Evidence

Hydra brute-force attempt (root)

Detection Evidence

Wazuh alerts - authentication failures

The attack generated multiple failed SSH login attempts, which were detected by the SIEM and mapped to brute-force techniques.

Phase 2 — Successful Attack (Valid Credentials)

  • Username used: siemadmin
  • Password discovered: 123456
  • Result: Successful authentication

Execution Evidence

Hydra successful login

Detection Evidence

Wazuh alerts - brute force + login success

Logs Generated

  • SSH authentication logs (failed attempts)
  • Successful login event
  • SIEM alerts triggered for brute-force activity

Detection

  • Detected by: Wazuh SIEM
  • MITRE ATT&CK: T1110.001 – Brute Force
  • MITRE ATT&CK: T1078 – Valid Accounts
  • Alert type: SSH authentication failure & brute-force detection

Result

  • Logs generated: ✔
  • Detected: ✔
  • Access gained: ✔
  • Blocked: ❌

Analysis:
The initial brute-force attempt failed due to incorrect username assumptions, despite the correct password being present in the wordlist. This highlights the importance of accurate reconnaissance in attack success.

After identifying the correct username, the attacker successfully gained access to the SIEM server using weak credentials. The activity was fully visible in the SIEM, with alerts generated for both failed attempts and successful authentication.

However, no account lockout or rate-limiting mechanisms were enforced, allowing the attacker to continue attempts until valid credentials were found. This represents a critical security gap in authentication protection.

The SSH service on SIEM01 was identified during the initial Nmap reconnaissance phase, and later targeted in this brute-force attack.

HTTP Flood Simulation (Web Server)

Description

A denial-of-service simulation was performed by generating a high volume of HTTP requests from the Kali Linux attacker machine against the internal web server. The goal was to evaluate the server’s resilience and the detection capabilities of the monitoring systems.

Execution

  • Tool used: Apache Benchmark (ab)
  • Command: ab -n 5000 -c 200 http://192.168.59.30/
  • Source machine: Kali Linux (192.168.100.10)
  • Target: Web Server (192.168.59.30)
  • Attack type: HTTP Flood (DoS Simulation)

Command Breakdown

  • ab (Apache Benchmark): A tool used to simulate HTTP load and evaluate web server performance under stress conditions.
  • -n 5000: Defines the total number of HTTP requests sent to the server, simulating high traffic volume.
  • -c 200: Specifies concurrency, meaning 200 requests are sent simultaneously to simulate multiple clients.
  • http://192.168.59.30/: Target web server receiving the HTTP requests.

Execution Evidence

HTTP flood execution (Kali)

Execution Findings

  • Thousands of HTTP requests were generated in a short time
  • The web server remained responsive and continued serving content
  • No visible performance degradation was observed from the client perspective

Web Server Response

Web server remains accessible

Logs Generated

  • Apache access logs recorded a high volume of incoming requests
  • All requests returned HTTP 200 (successful responses)
  • Traffic originated from the attacker machine (Kali)

Logs Evidence

Apache access logs (flood of requests)

Detection Evidence

Wazuh SIEM detection

Detection

  • Detected by: Suricata IDS
  • Alert type: TCP stream anomalies
  • Examples:
  • SURICATA STREAM invalid ACK
  • SYN resend with different sequence
  • Severity: Medium (Level 3)

Result

  • Service availability: Stable
  • Performance impact: Minimal
  • Requests handled successfully: ✔
  • Detected (Network Level): ✔
  • Detected (SIEM Correlation): ❌
  • Blocked: ❌

Analysis:
The HTTP flood simulation generated abnormal network traffic patterns, which were detected by Suricata as TCP stream anomalies, including invalid ACK packets and irregular TCP sequence behavior.

This indicates that the IDS was able to identify low-level network irregularities caused by the high volume of requests.

However, no high-level detection of HTTP flood or denial-of-service activity was observed, and no alerts were correlated to indicate an ongoing attack.

This highlights a gap in detection capabilities, where network anomalies are visible, but not properly contextualized into actionable security alerts.

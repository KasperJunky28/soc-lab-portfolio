Security Hardening & Defensive Transformation

Security Assessment Summary

The attack simulation demonstrated a full compromise of the lab environment, progressing from initial access to data exfiltration due to multiple security weaknesses across authentication, access control, and monitoring layers.

Key Security Gaps Identified

  • No account lockout or protection against brute-force attacks
  • Weak password policies enabling credential compromise
  • Overly permissive access to sensitive file shares
  • No monitoring of file access or data exfiltration
  • Lack of correlation between authentication and user activity
  • No active prevention mechanisms (rate limiting / blocking)

Impact

  • Unauthorized access to internal systems
  • Lateral movement across the network
  • Exposure and exfiltration of sensitive data

Conclusion:
The environment lacked layered security controls, allowing attackers to move freely within the network without being blocked or fully detected.

The following sections focus on implementing security controls to eliminate these weaknesses, improve detection visibility, and introduce prevention mechanisms.

1. Identity & Authentication Hardening

Overview

During the attack phase, weak authentication controls allowed successful brute-force attacks and unauthorized access to critical systems.

This phase focuses on strengthening authentication mechanisms to prevent unauthorized access and reduce the risk of credential compromise.

2. Network Hardening & Intrusion Prevention

Overview

During the attack phase, network-level protections were not enforced, allowing brute-force attacks and HTTP flood activity to proceed without restriction.

This phase focuses on implementing preventive controls to actively block malicious traffic and reduce the attack surface at the network level.

SSH Brute Force Protection (Fail2Ban)

Problem

The SSH service was exposed to unlimited authentication attempts, allowing brute-force attacks to succeed.

Implementation

  • Installed Fail2Ban on the SIEM server
  • Enabled and started the Fail2Ban service
  • Configured SSH protection (sshd jail)
  • Defined ban threshold and duration

Configuration Explanation

  • bantime: Defines how long an IP address is banned after exceeding the threshold. A longer ban time increases protection but may affect legitimate users.
  • findtime: The time window (in seconds) in which failed login attempts are counted. Helps define how quickly repeated attempts are considered malicious.
  • maxretry: The number of failed login attempts allowed before triggering a ban. Lower values provide stronger protection against brute-force attacks.
  • sshd jail: Specifies that Fail2Ban monitors SSH authentication logs and applies the configured rules to SSH login attempts.
  • logpath: Defines the log file Fail2Ban monitors (e.g., auth.log), which contains SSH authentication events.

Result

  • Brute-force protection enabled: ✔
  • Attacker IPs automatically banned: ✔

This control reduces the attack surface by limiting repeated authentication attempts, forcing attackers to change tactics or abandon the attack.

HTTP Flood Mitigation (iptables)

Problem

The web server was vulnerable to HTTP flood attacks, allowing high volumes of requests without any rate limiting or blocking.

Implementation

  • Implemented iptables rules for rate limiting
  • Configured connection thresholds per IP
  • Blocked excessive HTTP requests

Configuration Steps

iptables rule for HTTP flood protection

Show rule in iptables

Configuration Explanation

  • -A INPUT: Appends the rule to the INPUT chain, meaning it applies to incoming traffic to the server.
  • -p tcp --dport 80: Targets TCP traffic on port 80 (HTTP), focusing protection on the web server.
  • -m connlimit: Uses the connection limit module to restrict the number of simultaneous connections per IP address.
  • --connlimit-above: Defines the threshold for allowed connections. Connections exceeding this limit are considered suspicious.
  • -j DROP: Drops packets that exceed the defined threshold, effectively blocking excessive requests from the attacker.

This rule enforces rate limiting at the network level, preventing a single source from overwhelming the server with excessive HTTP requests.

Result

  • HTTP flood activity limited: ✔
  • Excessive requests blocked: ✔

Rate limiting prevents a single source from overwhelming the system, mitigating denial-of-service conditions.

3. Access Control & Visibility

Overview

During the attack phase, sensitive data was accessed due to overly permissive permissions and lack of monitoring on file activity.

This phase focuses on enforcing strict access control and improving visibility into file activity across the environment.

Security controls were implemented to restrict access based on least privilege principles, while auditing and monitoring mechanisms were introduced to detect unauthorized data access.

Security controls without visibility are ineffective — therefore, both access restriction and activity monitoring were implemented together.

File Server Hardening (Permissions)

Problem

Sensitive folders were accessible to a broad group of users due to overly permissive access control settings, including the presence of the "Users" group.

Risk

  • Unauthorized access to sensitive data
  • Increased risk of data exposure and exfiltration
  • Lack of proper access control enforcement

Implementation

  • Removed the "Users" group from sensitive folders
  • Restricted access based on specific roles
  • Applied least privilege access model

  • Unauthorized access removed: ✔
  • Access restricted based on roles: ✔
  • Least privilege successfully enforced: ✔

Analysis

By removing unnecessary permissions and enforcing least privilege, access to sensitive data was significantly reduced, limiting the potential impact of compromised accounts.

Event ID 4663 Explanation

Event ID 4663 is a Windows Security event that indicates an attempt to access an object, such as a file or folder, on the system.

  • What it represents: An access attempt to a specific object (file/folder)
  • Logged when: A user interacts with a resource that has auditing enabled
  • Includes: Username, object path, access type (read, write, delete), and process information

This event is generated only when both the global audit policy (Audit Object Access) and auditing on the specific resource are configured.

Event ID 4663 is critical for detecting unauthorized access and data exfiltration, as it provides detailed visibility into who accessed what resource and how.

Result

  • Visibility into user activity on sensitive data: ✔
  • Foundation established for SIEM-based detection: ✔

Analysis

File auditing requires both a global audit policy and specific auditing configuration on target resources. Without both components, no audit events would be generated.

This configuration bridges the gap between access control and detection, ensuring that all file access activity is logged and available for analysis.

4. Final Security Improvements Summary

Overview

Following the implementation of security hardening controls, the environment was significantly improved across authentication, network protection, access control, and monitoring capabilities.

The system evolved from a vulnerable, detection-limited environment into a layered security architecture with active prevention, visibility, and real-time detection.

Before vs After Comparison

Security Area Before Hardening After Hardening
Authentication Weak passwords, no lockout policy Strong password policy and account lockout enforced
Network Protection No protection against brute-force or DoS attacks Fail2Ban blocking brute-force + iptables limiting HTTP flood
Access Control Overly permissive access to sensitive shares Least privilege enforced, unauthorized access removed
Visibility No logging of file access File auditing enabled (Event ID 4663)
Detection No centralized monitoring or alerts Wazuh SIEM detecting file access and correlating events
Threat Detection No correlation between events Login detection + File Access correlation detecting data exfiltration

Key Security Improvements

  • Reduced attack surface by enforcing strong authentication controls
  • Introduced active network-level prevention mechanisms
  • Eliminated unauthorized access to sensitive data
  • Enabled full visibility into file access activity
  • Implemented real-time detection using SIEM
  • Established correlation-based detection for data exfiltration scenarios

Final Analysis

The environment transitioned from a vulnerable and reactive state into a layered security model combining prevention, visibility, and detection capabilities.

Authentication controls reduced the likelihood of initial compromise, network protections blocked external attacks, and access control mechanisms restricted lateral movement.

At the same time, auditing and SIEM integration enabled full visibility into user activity, allowing detection and correlation of suspicious behavior, including potential data exfiltration.

This demonstrates a complete security lifecycle: from attack simulation and vulnerability identification, to hardening implementation and validated detection.
